# 概要设计

## 1. 目标

系统目标是自动验证被测设备的扬声器是否有效。验证方式是通过 SDK 向被测设备发送扬声器测试音，由录音设备 A 录制现场复合流，再分析录像音频中是否包含该被测设备的测试音。

该方案面向实验室环境，允许存在环境噪声和多台设备同时播放音频。系统通过为每台被测设备生成差异较大的测试音，降低误判概率。

## 2. 分层结构

项目分为五层：

- SDK 封装层：`hikvision_voice.py`
- ISAPI 配置层：`hikvision_isapi.py`
- 测试音生成和发送层：`use_cases/voice_talk_cases.py`
- 扬声器测试编排层：`use_cases/speaker_test_cases.py`
- 录像音频分析层：`video_analysis.py`

命令行入口：

- `demo_speaker_test.py`
- `demo_video_analysis.py`
- `demo_stream_record.py`
- `demo_composite_stream_record.py`
- `demo_random_audio_talk_use_case.py`

## 3. 核心流程

扬声器测试流程：

1. 初始化两个 SDK 实例，一个用于向被测设备发送音频，一个用于录音设备取流录像。
2. 登录被测设备。
3. 检查被测设备是否支持 MicIn 能力。
4. 登录录音设备 A。
5. 将录音设备 A 配置为 MicIn，并确保复合流携带音频。
6. 将被测设备配置为 MicIn 输入、Speaker 输出，并调高音量。
7. 根据被测设备生成扬声器测试音。
8. 启动录音设备 A 的复合流录像。
9. 录像预录 `2s`。
10. 通过语音转发向被测设备发送 `4s` 测试音。
11. 播放结束后继续录制 `3s`。
12. 依次调用 `NET_DVR_StopSaveRealData` 和 `NET_DVR_StopRealPlay`。
13. 等待 `3s` 后检查录像文件。
14. 从录像中提取音频。
15. 判断是否有声音。
16. 判断是否包含当前被测设备的扬声器测试音。
17. 输出最终结论和日志。

## 4. 扬声器测试音设计

测试音由 `VoiceTalkUseCases` 生成，采样格式为：

- 单声道
- `8000Hz`
- `pcm_s16le`

默认测试音不是固定 `1234`，而是根据 `test-tone-id` 生成稳定音频。`demo_speaker_test.py` 默认使用被测设备 IP 作为 `test-tone-id`。

测试音差异维度：

- 数字序列：默认 6 位。
- 音调比例：根据 `test-tone-id` 稳定变化。
- 数字间隔：根据 `test-tone-id` 稳定变化。
- 有效发声占比：根据 `test-tone-id` 稳定变化。

手动参数：

- `--test-tone-id`：指定设备测试音 ID。
- `--digit-sequence`：指定固定数字序列。
- `--seed`：按 seed 生成稳定数字序列。

## 5. 判定逻辑

最终判定由 `RecordedVideoAnalyzer.detect_reference_audio(...)` 完成。

分析步骤：

1. 使用 `ffmpeg` 从 mp4 提取音频。
2. 转成 `8000Hz / mono / pcm_s16le`。
3. 按 `20ms` 切帧。
4. 计算 RMS 判断是否存在有效声音。
5. 计算 RMS 能量签名。
6. 计算 DTMF 频率能量签名。
7. 对参考音频有效发声段做裁剪。
8. 执行滑动窗口相关性匹配。
9. 根据期望数字序列执行频偏容忍模板搜索。
10. 取综合最高分作为 `score`。

判定条件：

- `has_sound=True`
- `match=True`
- `score >= similarity_threshold`

默认阈值为 `0.8`。

## 6. 录像关闭设计

实测验证后，录像关闭统一采用显式两步：

1. `NET_DVR_StopSaveRealData`
2. `NET_DVR_StopRealPlay`

两步完成后等待 `3s`，再检查录像文件大小并继续分析。

该顺序已同步到：

- `StreamRecorder.stop()`
- `demo_stream_record.py`
- `demo_composite_stream_record.py`
- `SpeakerTestUseCases`

## 7. 日志和输出

扬声器测试会在对应被测设备 IP 目录下生成日志：

```text
recordings/speaker_tests/<device-ip>/speaker_test_log_<ip>_<timestamp>.log
```

日志同步控制台输出，每行带执行时间。最终结论包含：

- 录像路径
- 参考音频路径
- 数字序列
- `has_sound`
- `match`
- `score`
- `threshold`

## 8. 扩展方向

后续如果需要更强抗干扰能力，可以扩展：

- 为每台设备生成更长数字序列。
- 增加非 DTMF 频段的专属水印音。
- 对多个候选设备的测试音同时评分，输出最高分和次高分差距。
- 增加环境噪声基线估计和信噪比日志。
