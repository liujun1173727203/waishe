# 概要设计

## 1. 目标

系统目标是自动验证被测设备的扬声器是否有效。验证方式是通过 SDK 向被测设备发送扬声器测试音，由录音设备 A 录制现场复合流，再分析录像音频中是否包含该被测设备的测试音。

该方案面向实验室环境，允许存在环境噪声和多台设备同时播放音频。系统通过为每台被测设备生成差异较大的测试音，降低误判概率。

所有完整测试入口都会把 `audioCompressionType` 作为测试维度。默认读取设备能力中的压缩类型选项并逐个执行，录像文件、参考音频和提取音频的文件名都会包含当前 `audioCompressionType`。

## 3. 核心流程

扬声器测试流程：

1. 初始化两个 SDK 实例，一个用于向被测设备发送音频，一个用于录音设备取流录像。
2. 登录被测设备。
3. 从录音设备池申请录音设备租约，池内设备达到 `max_connections` 时按先来先服务等待，默认最多等待 `300s`。
4. 检查被测设备是否支持 MicIn 能力。
5. 登录租约分配的录音设备 A。
6. 将录音设备 A 配置为 MicIn，并确保复合流携带音频。
7. 将被测设备配置为 MicIn 输入、Speaker 输出，并调高音量。
8. 根据被测设备生成扬声器测试音。
9. 启动录音设备 A 的复合流录像。
10. 录像预录 `2s`。
11. 通过语音转发向被测设备发送 `4s` 测试音。
12. 播放结束后继续录制 `3s`。
13. 依次调用 `NET_DVR_StopSaveRealData` 和 `NET_DVR_StopRealPlay`。
14. 等待 `3s` 后检查录像文件。
15. 从录像中提取音频。
16. 判断是否有声音。
17. 判断是否包含当前被测设备的扬声器测试音。
18. 释放录音设备池租约，输出最终结论和日志。

LineOut 接入测试流程复用扬声器测试，配置差异为：

- 被测设备 `audioOutputType=LineOut`。
- 被测设备 LineOut 端接外置扬声器。
- 默认遍历被测设备支持的 `audioCompressionType`。
- 入口为 `demo_lineout_test.py`，也可通过 `demo_speaker_test.py --test-device-output-type LineOut` 执行。

拾音器测试流程：

1. 登录测试设备。
2. 检查并配置测试设备音频输入、输出和最大音量。
3. 登录播放设备 A。
4. 配置播放设备 A 为 `MicIn + Speaker` 并调高音量。
5. 优先开启测试设备复合流录像。
6. 播放设备 A 通过对讲播放 `4s` 测试音。
7. 测试设备继续录制 `3s` 后关闭录像。
8. 如果复合流录像可用，则分析测试设备录像中是否包含播放设备 A 的测试音。
9. 如果复合流不支持、录像启动失败或录像文件为空，则 fallback 到测试设备对讲回调音频。
10. 对讲回调 fallback 会保存测试设备接收到的 PCM WAV，并分析其中是否包含播放设备 A 的测试音。

LineIn 接入测试流程复用拾音器测试，配置差异为：

- 测试设备 `audioInputType=LineIn`。
- 测试设备 LineIn 端接外置拾音器。
- 默认遍历播放设备 A 支持的 `audioCompressionType`。
- 同样支持复合流录像优先、对讲回调音频 fallback。
- 入口为 `demo_linein_test.py`，也可通过 `demo_pickup_test.py --test-device-input-type LineIn` 执行。

## 4. 扬声器测试音设计

测试音由 `VoiceTalkUseCases` 生成，采样格式为：

- 单声道
- `8000Hz`
- `pcm_s16le`

默认测试音根据 `test-tone-id` 生成稳定的离散频率音频。`demo_speaker_test.py` 默认使用被测设备 IP 作为 `test-tone-id`。

测试音差异维度：

- `frequency_profile`：默认 8 段频率。
- 频率范围：默认 `900Hz ~ 3200Hz`，适配 `8000Hz` 对讲采样链路。
- 频率跨度：相邻频段尽量拉开，减少多设备同时播放时的误判。
- 频率过渡：段边界使用短淡入淡出，避免硬切爆音。

手动参数：

- `--test-tone-id`：指定设备测试音 ID。
- `--seed`：参与离散频率指纹生成，保证可复现。

## 5. 判定逻辑

最终判定由 `RecordedVideoAnalyzer.detect_reference_audio(...)` 完成。

分析步骤：

1. 使用 `ffmpeg` 从 mp4 提取音频。
2. 转成 `8000Hz / mono / pcm_s16le`。
3. 按 `20ms` 切帧。
4. 计算 RMS 判断是否存在有效声音。
5. 从参考测试音提取频率指纹探针。
6. 在录像音频中按时间窗口搜索这些频率分量。
7. 计算目标频率相对背景频率的能量强度。
8. 综合频率存在度、覆盖率和时间窗口，输出 `score`。

判定条件：

- `has_sound=True`
- `match=True`
- `score >= similarity_threshold`

默认阈值为 `0.7`。

## 6. 录像关闭设计

实测验证后，录像关闭统一采用显式两步：

1. `NET_DVR_StopSaveRealData`
2. `NET_DVR_StopRealPlay`

两步完成后等待 `3s`，再检查录像文件大小并继续分析。

该顺序已同步到：

- `StreamRecorder.stop()`
- `demo_stream_record.py`
- `demo_composite_stream_record.py`
- `SpeakerTestUseCases`

## 7. 抓图设计

SDK 封装提供三种抓图入口：

- `capture_jpeg_picture(...)`：只调用 `NET_DVR_CaptureJPEGPicture`。
- `capture_stream_picture(...)`：启动短时预览，调用 `NET_DVR_CapturePicture`。
- `capture_picture(...)`：默认入口，先 JPEG 抓图，失败后自动 fallback 到取流抓图。

设计约束：

- 默认优先 JPEG，因为成功时直接输出 `.jpg`。
- 部分设备不支持 JPEG 抓图时，不终止用例，而是自动走取流抓图。
- 取流抓图使用 SDK 预览抓图接口，输出 BMP；如果用户传入 `.jpg`，文件名会自动改为 `_stream.bmp`。

## 8. 日志和输出

扬声器测试会在对应被测设备 IP 目录下生成日志：

```text
recordings/speaker_tests/<device-ip>/speaker_test_log_<ip>_<timestamp>.log
```

日志同步控制台输出，每行带执行时间。最终结论包含：

- 录像路径
- 参考音频路径
- 离散频率指纹 `frequency_profile`
- `has_sound`
- `match`
- `score`
- `threshold`

补光灯测试输出到：

```text
recordings/supplement_light_tests/<device-ip>/
```

包含：

- 执行日志。
- baseline 抓图。
- `0 / 50 / 100` 三挡抓图。
- 每档中心 ROI 亮度均值。
- 开启判定、档位递增判定和最终结论。

## 9. 补光灯测试设计

补光灯测试目标是验证：

- 补光灯开启后画面亮度有明显变化。
- 补光灯强度 `0 / 50 / 100` 三挡调节有效。

核心流程：

1. 查询 `/ISAPI/Image/channels/<channel>/SupplementLight/capabilities`。
2. 查询 `/ISAPI/Image/channels/<channel>/SupplementLight` 当前配置。
3. 设置补光灯关闭或强度 `0`，抓图作为 baseline。
4. 设置补光灯开启，并依次设置强度 `0 / 50 / 100`。
5. 每档等待稳定后抓图。
6. 计算图片中心 ROI 灰度均值。
7. 判断 `100` 档相对 baseline 是否超过开启阈值。
8. 判断 `50` 档相对 `0` 档、`100` 档相对 `50` 档是否递增且超过阈值。

抓图复用 SDK 抓图封装，默认先 JPEG 抓图，失败后自动 fallback 到取流抓图。

## 10. 扩展方向

后续如果需要更强抗干扰能力，可以扩展：

- 为每台设备生成更长或更多段的离散频率指纹。
- 增加频率跳变约束、二次谐波校验或专属水印音。
- 对多个候选设备的测试音同时评分，输出最高分和次高分差距。
- 增加环境噪声基线估计和信噪比日志。
