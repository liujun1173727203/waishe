# 使用文档

## 1. 项目用途

本项目用于通过海康 SDK 和 ISAPI 自动验证设备扬声器是否有效。完整测试流程是：向被测设备发送一段专属扬声器测试音，由录音设备 A 录制复合流，随后分析录像音频中是否包含被测设备的测试音。

该设计适用于实验室中多台设备可能同时播放声音的场景。每台被测设备默认根据自身 IP 生成稳定且差异较大的测试音，用于降低环境噪声和其他设备播放音频带来的误判。

## 2. 环境要求

- Windows 64 位。
- Python 3。
- 项目自带 `libs/win64` 下的 `HCNetSDK` 依赖。
- 需要本机安装 `ffmpeg`，或通过参数指定 `ffmpeg.exe`。

默认 `ffmpeg` 路径：

```text
D:\ffmpeg\ffmpeg-2026-05-28-git-7b46c6a2a3-essentials_build\bin\ffmpeg.exe
```

## 3. 扬声器测试

基础命令：

```powershell
python .\demo_speaker_test.py `
  --host 10.18.117.22 `
  --port 8000 `
  --username admin `
  --password asdf!234 `
  --recorder-pool-config .\configs\recorder_device_pool.json
```

常用参数：

- `--host`：被测设备 IP，即要验证扬声器的设备。
- `--voice-channel`：被测设备语音通道，默认 `2`。
- `--recorder-pool-config`：录音设备池配置文件，默认 `configs/recorder_device_pool.json`。
- `--recorder-pool-wait-seconds`：录音设备池最大等待时间，默认 `300s`。
- `--send-duration`：测试音时长，默认 `4s`。
- `--similarity-threshold`：匹配阈值，默认 `0.7`。
- `--test-tone-id`：扬声器测试音 ID，默认使用 `--host`。
- `--audio-compression-types`：音频压缩类型遍历列表，默认 `auto`，表示按设备能力中的 `audioCompressionType` 选项逐个测试。
- `--ffmpeg-path`：指定 `ffmpeg.exe` 路径。
- `--enable-log`：开启 SDK 日志。

默认会遍历设备支持的所有 `audioCompressionType`。如果只想测试指定类型，可以传逗号分隔列表：

```powershell
python .\demo_speaker_test.py --audio-compression-types G.711ulaw,G.711alaw,PCM
```

录音设备池配置示例：

```json
{
  "devices": [
    {
      "id": "recorder-a-10.40.230.23",
      "host": "10.40.230.23",
      "port": 8000,
      "username": "admin",
      "password": "asdf!234",
      "channel": 0,
      "voice_channel": 1,
      "max_connections": 1
    }
  ]
}
```

选择录音设备时只能从池中选择。如果所有池内设备连接数均达到 `max_connections`，后续测试进入等待队列，遵守先来先服务策略，默认最多等待 `5` 分钟。

### 3.1 LineOut 外接扬声器测试

`LineOut` 用例与扬声器测试流程一致，区别是被测设备配置为 `audioOutputType=LineOut`。被测设备需要在 LineOut 端接外置扬声器，录音设备 A 仍负责录像和采集现场音频。

```powershell
python .\demo_lineout_test.py `
  --host 10.18.117.22 `
  --username admin `
  --password asdf!234 `
  --recorder-pool-config .\configs\recorder_device_pool.json
```

等价命令：

```powershell
python .\demo_speaker_test.py --test-device-output-type LineOut
```

## 4. 拾音器测试

拾音器测试由测试设备自己录像，设备 A 开启对讲并播放测试音。系统分析测试设备录像中是否包含设备 A 播出的测试音，用于判断测试设备拾音是否有效。

```powershell
python .\demo_pickup_test.py `
  --host 10.18.117.22 `
  --recorder-pool-config .\configs\recorder_device_pool.json `
  --playback-host 10.40.230.23 `
  --playback-username admin `
  --playback-password asdf!234
```

拾音器测试由测试设备自己录像，因此 `--host` 指定的测试设备也必须存在于录音设备池中，并受 `max_connections` 约束。

拾音器测试同样默认遍历播放设备 A 支持的所有 `audioCompressionType`。如果只想测试指定类型：

```powershell
python .\demo_pickup_test.py --audio-compression-types G.711ulaw,G.711alaw,PCM
```

拾音器测试默认优先使用测试设备复合流录像分析。如果测试设备不支持复合流音频、录像启动失败或录像文件为空，会自动 fallback 到测试设备对讲回调音频：

```text
analysis_source=recording       # 使用录像音频分析
analysis_source=talk_callback   # 使用测试设备对讲回调音频分析
```

fallback 时会保存：

```text
pickup_test_talk_callback_audio_<ip>_<audioCompressionType>_<timestamp>.wav
```

### 4.1 LineIn 外接拾音器测试

`LineIn` 用例与拾音器测试流程一致，区别是测试设备配置为 `audioInputType=LineIn`。测试设备需要在 LineIn 端接外置拾音器，设备 A 仍通过扬声器播放测试音。

LineIn 同样支持复合流录像优先、对讲回调音频 fallback。

```powershell
python .\demo_linein_test.py `
  --host 10.18.117.22 `
  --recorder-pool-config .\configs\recorder_device_pool.json `
  --playback-host 10.40.230.23 `
  --playback-username admin `
  --playback-password asdf!234
```

等价命令：

```powershell
python .\demo_pickup_test.py --test-device-input-type LineIn
```

## 5. 测试时间线

默认时间线如下：

1. 录音设备 A 先启动复合流录像。
2. 录像启动后等待 `2s`，保留播放前环境音。
3. 被测设备播放 `4s` 扬声器测试音。
4. 播放结束后继续录制 `3s`。
5. 调用 `NET_DVR_StopSaveRealData` 停止保存录像数据。
6. 调用 `NET_DVR_StopRealPlay` 停止实时预览。
7. 等待 `3s`，再检查录像文件大小并进入音频分析。

日志中会看到类似：

```text
[speaker-test] expected record timeline pre_talk=2.00s talk=4.00s post_talk=3.00s total>=8.00s
[speaker-test] keep recorder device A recording for 2.00s before talk starts
[speaker-test] recorder device A stream stopped, wait 3.00s before file flush check
```

## 6. 扬声器测试音

默认根据被测设备生成稳定的离散频率测试音。测试音包含：

- 设备专属 `frequency_profile`，默认在 `900Hz ~ 3200Hz` 之间分布。
- 多段固定正弦频率，相邻频段跨度较大。
- 段边界使用短淡入淡出，避免硬切爆音。
- 默认 `4s` 发声时长，适配对讲链路的 `8000Hz` 采样。

这不是为了定位多台设备中的任意音源，而是为了判断“当前被测设备是否播放出了它自己的测试音”。当实验室内多台设备同时播放时，该设计可以减少其他设备音频导致的误判。

示例：

```powershell
python .\demo_speaker_test.py --host 10.18.117.22 --test-tone-id 10.18.117.22
```

## 7. 输出文件

默认输出目录：

```text
recordings/speaker_tests/<device-ip>/
```

文件说明：

- `speaker_test_log_<ip>_<timestamp>.log`：完整执行日志，每行带时间戳。
- `speaker_test_reference_<ip>_<audioCompressionType>_<timestamp>_*.wav`：本次生成的扬声器测试音。
- `speaker_test_record_<ip>_<audioCompressionType>_<timestamp>.mp4`：录音设备 A 录制的录像。
- `speaker_test_record_audio_<ip>_<audioCompressionType>_<timestamp>.wav`：从录像中提取的音频。

拾音器测试输出到：

```text
recordings/pickup_tests/<device-ip>/
```

拾音器文件名同样包含 `<audioCompressionType>`，例如：

```text
pickup_test_record_10_18_117_22_G.711ulaw_20260603_101010.mp4
```

最终日志示例：

```text
[speaker-test] match analysis done matched=True score=0.9123 threshold=0.70
[speaker-test] final conclusion record=... reference=... frequency_profile=1479.7,3024.4,1138.7,2883.0,1651.4,2524.3,1527.7,2693.5 has_sound=True match=True score=0.9123 threshold=0.70
```

其中：

- `has_sound=True` 表示录像音频中存在有效声音。
- `match=True` 表示录到了当前被测设备的扬声器测试音。
- `score` 是最终匹配分数。
- `threshold` 是判定阈值。
- `frequency_profile` 是本次离散频率测试音的设备指纹。
- 默认离散频率音以频率指纹分量检测为主。

## 8. 单独分析录像

可以脱离完整用例，单独分析某个录像：

```powershell
python .\demo_video_analysis.py `
  --video .\recordings\speaker_tests\10.18.117.22\speaker_test_record_10_18_117_22_20260602_152906.mp4 `
  --reference-audio .\recordings\speaker_tests\10.18.117.22\speaker_test_reference_10_18_117_22_20260602_152906_20260602_152907.wav `
  --score-threshold 0.7
```

单独分析时只需要传入 `--reference-audio`。

## 9. 普通录像

```powershell
python .\demo_stream_record.py `
  --host 192.168.1.64 `
  --recorder-pool-config .\configs\recorder_device_pool.json `
  --duration 10
```

普通录像中 `--host` 只用于从录音设备池中选择对应设备，登录用户名和密码来自池配置。

关闭录像时同样执行：

1. `NET_DVR_StopSaveRealData`
2. `NET_DVR_StopRealPlay`
3. 等待 `3s`

## 10. 复合流录像

```powershell
python .\demo_composite_stream_record.py `
  --host 192.168.1.64 `
  --recorder-pool-config .\configs\recorder_device_pool.json `
  --duration 10
```

默认行为：

1. 登录设备。
2. 调整音频输入和输出音量。
3. 查询复合流能力。

6. 按统一顺序停止录像。

## 11. 抓图

默认抓图优先使用设备 JPEG 抓图接口。如果设备不支持 JPEG 抓图或 JPEG 抓图失败，会自动启动短时取流并通过预览句柄抓图。

```powershell
python .\demo_capture_picture.py `
  --host 10.18.117.22 `
  --username admin `
  --password asdf!234
```

抓图模式：

- `--mode auto`：默认模式，先 JPEG 抓图，失败后自动走取流抓图。
- `--mode jpeg`：只使用 JPEG 抓图，失败直接报错。
- `--mode stream`：只使用取流抓图。

输出说明：

- JPEG 抓图成功时，输出 `.jpg` 文件。
- 取流抓图使用 SDK 预览抓图接口，输出 BMP 内容；如果传入 `.jpg` 路径，会自动改成同名 `_stream.bmp`，避免扩展名和实际格式不一致。
- 默认目录为 `recordings/captures/<device-ip>/`。

## 12. 补光灯测试

补光灯测试会通过 ISAPI 控制补光灯，并通过抓图亮度判断补光灯是否真实开启、强度是否随 `0 / 50 / 100` 三挡递增。

```powershell
python .\demo_supplement_light_test.py `
  --host 10.18.117.22 `
  --username admin `
  --password asdf!234
```

默认流程：

1. 查询 `/ISAPI/Image/channels/<channel>/SupplementLight/capabilities` 和当前配置。
2. 设置补光灯关闭或强度 `0`，等待稳定后抓图，作为 baseline。
3. 设置补光灯开启，依次设置强度 `0`、`50`、`100`。
4. 每档等待 `--settle-seconds` 后抓图。
5. 使用 `ffmpeg` 读取中心 `40% x 40%` ROI 的灰度均值。
6. 判断 `100` 档相对 baseline 是否超过 `--on-threshold`。
7. 判断 `50 > 0`、`100 > 50` 是否分别超过 `--level-threshold`。

常用参数：

- `--channel`：图像通道，`0` 表示自动使用 SDK 登录返回的默认预览通道。
- `--settle-seconds`：每次设置补光灯后等待稳定时间，默认 `2s`。
- `--on-threshold`：开启补光灯亮度差阈值，默认 `10`。
- `--level-threshold`：相邻档位亮度差阈值，默认 `5`。
- `--ffmpeg-path`：指定 `ffmpeg.exe` 路径。

输出目录：

```text
recordings/supplement_light_tests/<device-ip>/
```

输出文件包括：

- `supplement_light_test_log_<ip>_<timestamp>.log`
- `supplement_light_baseline_off_<ip>_<timestamp>.jpg`
- `supplement_light_level_0_<ip>_<timestamp>.jpg`
- `supplement_light_level_50_<ip>_<timestamp>.jpg`
- `supplement_light_level_100_<ip>_<timestamp>.jpg`

如果 JPEG 抓图失败，会自动 fallback 到取流抓图，文件可能保存为 `_stream.bmp`。

## 13. 常见问题

如果没有音频分析结论，优先检查：

- `ffmpeg` 路径是否正确。
- 日志中是否出现 `analysis dependency ready ffmpeg=...`。
- 录音设备是否生成了非空 mp4。
- 是否出现 `sound analysis done ...`。
- 是否出现 `match analysis done ...`。

如果 `has_sound=True` 但 `match=False`：

- 说明录像中有声音，但没有达到当前被测设备测试音的匹配阈值。
- 可检查录音设备 A 和被测设备 B 的距离、环境噪声、扬声器音量、麦克风音量。
- 可使用 `--test-tone-id` 复现同一设备的离散频率测试音。
