# 详细设计

## 1. 文件结构

### 1.1 `hikvision_voice.py`

负责 HCNetSDK 的 Python 封装：

- SDK 初始化、日志、清理。
- 设备登录和登出。
- 语音转发。
- 实时预览取流。
- 保存录像文件。
- 显式关闭录像保存和实时预览。
- JPEG 抓图和取流抓图。

关键类：

- `HikvisionVoiceSDK`
- `DeviceSession`
- `VoiceForwardSession`
- `StreamRecorder`
- `CapturePictureResult`

### 1.2 `hikvision_isapi.py`

负责 ISAPI 配置：

- 查询双向音频能力。
- 查询 `audioInputType` 和 `audioOutputType` 的能力选项。
- 查询 `audioCompressionType` 的能力选项。
- 设置 `MicIn` 或 `LineIn` 输入。
- 设置 `Speaker` 或 `LineOut` 输出。
- 设置当前测试轮次的 `audioCompressionType`。
- 调整麦克风和扬声器音量。
- 查询和开启复合流音频。

关键类：

- `HikvisionIsapiClient`
- `AudioInputCapabilityStatus`
- `TwoWayAudioChannelStatus`
- `CompositeStreamStatus`

### 1.3 `use_cases/voice_talk_cases.py`

负责扬声器测试音生成和发送：

- 根据 `test-tone-id` 生成稳定连续频率指纹。
- 生成 `900Hz ~ 3200Hz` 连续频率音频，兼容旧版 DTMF 数字音调试模式。
- 按设备编码格式转为 `G.711u`、`G.711a` 或 `PCM`。
- 按 `20ms` 拆分发送帧。
- 通过语音转发接口发送到被测设备。

### 1.4 `use_cases/speaker_test_cases.py`

负责编排完整扬声器测试：

- 配置被测设备和录音设备。
- 启动录音设备复合流录像。
- 发送被测设备扬声器测试音。
- 停止录像并等待落盘。
- 分析录像音频。
- 输出结构化结果。

`LineOut` 接入测试复用该编排逻辑，通过 `test_device_output_type="LineOut"` 将被测设备配置为 LineOut 输出。

### 1.5 `use_cases/pickup_test_cases.py`

负责编排完整拾音器测试：

- 配置测试设备音频输入、输出和复合流录像。
- 配置播放设备 A。
- 让播放设备 A 通过扬声器播放测试音。
- 优先分析测试设备录像音频。
- 复合流录像不可用时 fallback 到测试设备对讲回调音频。
- 输出结构化结果。

`LineIn` 接入测试复用该编排逻辑，通过 `test_device_input_type="LineIn"` 将测试设备配置为 LineIn 输入。

### 1.6 `use_cases/supplement_light_cases.py`

负责编排完整补光灯测试：

- 查询补光灯能力和当前配置。
- 设置 baseline 状态并抓图。
- 设置补光灯开启。
- 依次设置强度 `0 / 50 / 100`。
- 每档抓图并计算中心 ROI 亮度。
- 输出开启判定、强度递增判定和最终结论。

### 1.7 `video_analysis.py`

负责录像音频分析：

- 调用 `ffmpeg` 提取音频。
- 判断是否有有效声音。
- 计算参考测试音与录像音频匹配分数。
- 支持音调偏移、环境噪声和多设备同时播放下的连续频率指纹匹配。

## 2. SDK 调用设计

### 2.1 初始化和清理

初始化：

1. 加载 `HCNetSDK.dll`。
2. 配置 `HCNetSDKCom` 路径。
3. 配置 OpenSSL 依赖路径。
4. 调用 `NET_DVR_Init`。
5. 可选调用 `NET_DVR_SetLogToFile`。

清理：

1. 清理 Python 侧 callback 引用。
2. 调用 `NET_DVR_Cleanup`。

### 2.2 登录和登出

登录接口：

- `NET_DVR_Login_V40`

登出接口：

- `NET_DVR_Logout`

登出必须在实时预览、保存录像、语音转发关闭之后执行。

### 2.3 语音转发

启动：

- `NET_DVR_StartVoiceCom_MR_V30`

发送：

- `NET_DVR_VoiceComSendData`

停止：

- `NET_DVR_StopVoiceCom`

callback 生命周期策略：

- 启动成功后保存 Python callback 引用。
- SDK stop 成功后再释放 callback 引用。

### 2.4 实时录像

启动：

1. 填充 `NET_DVR_PREVIEWINFO`。
2. 调用 `NET_DVR_RealPlay_V40`。
3. 调用 `NET_DVR_SaveRealData`。
4. 保存实时数据 callback 引用。
5. 等待首帧数据，防止空录像。

关闭：

1. 调用 `NET_DVR_StopSaveRealData`。
2. 调用 `NET_DVR_StopRealPlay`。
3. 释放 callback 引用。
4. 等待 `3s`。
5. 检查录像文件大小。

该关闭顺序经过 `2026-06-02` 实测验证，已同步到所有录像入口。

### 2.5 抓图

JPEG 抓图：

1. 填充 `NET_DVR_JPEGPARA`。
2. 调用 `NET_DVR_CaptureJPEGPicture`。
3. 检查输出文件是否存在且非空。

取流抓图：

1. 填充 `NET_DVR_PREVIEWINFO`。
2. 调用 `NET_DVR_RealPlay_V40` 启动短时预览。
3. 等待首批流数据，最多等待 `5s`。
4. 调用 `NET_DVR_CapturePicture`。
5. 调用 `NET_DVR_StopRealPlay`。
6. 释放实时数据 callback 引用。
7. 检查输出文件是否存在且非空。

默认入口 `capture_picture(...)`：

1. 先调用 JPEG 抓图。
2. 如果 JPEG 抓图失败，自动调用取流抓图。
3. 返回 `CapturePictureResult`，其中包含实际文件路径、抓图方式和是否 fallback。

取流抓图输出 BMP 内容。如果传入 `.jpg` 路径，封装会自动改成同名 `_stream.bmp`，避免文件扩展名和真实格式不一致。

## 3. ISAPI 配置设计

### 3.1 双向音频能力

能力接口：

- `GET /ISAPI/System/TwoWayAudio/channels/capabilities`

用途：

- 读取 `audioInputType` 的 `opt` 选项，例如 `MicIn`、`LineIn`。
- 读取 `audioOutputType` 的 `opt` 选项，例如 `Speaker`、`LineOut`。
- 读取 `audioCompressionType` 的 `opt` 选项，例如 `G.711ulaw`、`G.711alaw`、`PCM`。
- 保留 `MicIn` 和 `Speaker` 的兼容性字段，供旧逻辑继续使用。

### 3.2 双向音频通道配置

配置接口：

- `GET /ISAPI/System/TwoWayAudio/channels/<channel>`
- `PUT /ISAPI/System/TwoWayAudio/channels/<channel>`

配置目标：

- `enabled=true`
- `audioInputType=MicIn` 或 `LineIn`
- `audioOutputType=Speaker` 或 `LineOut`
- `audioCompressionType=<当前遍历类型>`
- 麦克风音量最大
- 扬声器音量最大

### 3.3 复合流音频配置

接口：

- `GET /ISAPI/Streaming/channels/<trackStreamID>/capabilities`
- `GET /ISAPI/Streaming/channels/<trackStreamID>`
- `PUT /ISAPI/Streaming/channels/<trackStreamID>`

目标：

- 确认通道支持复合流录音。
- 确保 `Audio.enabled` 或 `Audio.enable` 为 `true`。

### 3.4 补光灯配置

优先接口：

- `GET /ISAPI/Image/channels/<channel>/SupplementLight/capabilities`
- `GET /ISAPI/Image/channels/<channel>/SupplementLight`
- `PUT /ISAPI/Image/channels/<channel>/SupplementLight`

配置字段存在型号差异，代码会兼容以下常见字段：

- 开关字段：`enabled`、`enable`
- 模式字段：`mode`、`supplementLightMode`、`lightMode`
- 强度字段：`brightness`、`supplementLightIntensity`、`supplementLightBrightness`、`intensity`、`lightBrightness`、`whiteLightBrightness`

设置策略：

1. 先 GET 当前配置。
2. 只修改已存在字段；如果字段不存在则创建候选字段中的第一个字段。
3. PUT 回原配置路径。
4. 再 GET 刷新状态。

## 4. 扬声器测试音设计

### 4.1 生成输入

`VoiceTalkUseCases.prepare_random_audio_file(...)` 支持：

- `duration_seconds`：默认 `4s`。
- `digit_sequence`：可选固定数字序列；仅传入该参数时启用旧版 DTMF 兼容模式。
- `fingerprint_source`：测试音 ID，扬声器用例默认传被测设备 IP。
- `seed`：可选随机种子。
- `amplitude_ratio`：振幅比例。

### 4.2 默认测试音

如果未指定 `digit_sequence`，系统根据 `fingerprint_source` 生成连续频率指纹。生成逻辑使用 SHA-256，保证同一测试音 ID 生成结果稳定，不同测试音 ID 的结果差异较大。

默认连续频率音包含：

- `frequency_profile`：默认 8 段频率。
- `min_frequency=900Hz`。
- `max_frequency=3200Hz`。
- 段间平滑过渡，声音连续。
- 相邻频段按低/高区间交替选择，尽量拉大跨度。

这些维度可以帮助区分当前设备的测试音和其他设备同时播放的干扰音。

### 4.3 音频格式

内部生成格式：

- 单声道
- `8000Hz`
- `pcm_s16le`

默认测试音使用连续正弦频率合成，不再按数字拆分。手动传入 `--digit-sequence` 时，才使用旧版 DTMF 双音频率合成：

- 低频组：`697`、`770`、`852`、`941`
- 高频组：`1209`、`1336`、`1477`

发送前根据设备当前音频编码转换为：

- `G.711u`
- `G.711a`
- `PCM`

## 5. 扬声器测试流程

### 5.1 输入参数

`SpeakerTestUseCases.run_speaker_test(...)` 主要参数：

- `session`
- `record_channel`
- `voice_channel`
- `record_duration_seconds`
- `send_duration_seconds`
- `similarity_threshold`
- `amplitude_ratio`
- `seed`
- `digit_sequence`
- `fingerprint_source`
- `test_device_output_type`：默认 `Speaker`，LineOut 接入测试传 `LineOut`。
- `audio_compression_type`：当前轮次使用的 `audioCompressionType`。

### 5.2 执行步骤

1. 检查参数合法性。
2. 检查 `ffmpeg` 是否可用。
3. 设置 SDK 对讲模式为库模式。
4. 检查被测设备 MicIn 能力。
5. 登录录音设备 A。
6. 配置录音设备 A 为 MicIn，并确保复合流携带音频。
7. 配置被测设备为 `MicIn` 输入、指定输出和当前 `audioCompressionType`。
8. 生成当前被测设备的扬声器测试音。
9. 启动录音设备 A 复合流录像。
10. 等待首帧数据。
11. 预录 `2s`。
12. 向被测设备发送 `4s` 测试音。
13. 播放结束后继续录制 `3s`。
14. 显式停止保存录像。
15. 显式停止实时预览。
16. 等待 `3s`。
17. 检查录像文件大小。
18. 登出录音设备 A。
19. 提取录像音频。
20. 判断是否有声音。
21. 匹配当前被测设备的扬声器测试音。
22. 输出最终结论。

### 5.3 输出结果

`SpeakerTestResult` 字段：

- `login_verified`
- `audio_input_status`
- `two_way_audio_status`
- `recorder_audio_status`
- `composite_status`
- `record_file_path`
- `reference_audio_path`
- `extracted_audio_path`
- `talk_result`
- `sound_result`
- `match_result`

### 5.4 LineOut 接入测试

入口：

```powershell
python .\demo_lineout_test.py
```

实现方式：

- 复用 `SpeakerTestUseCases.run_speaker_test(...)`。
- 默认注入 `test_device_output_type="LineOut"`。
- 能力校验要求被测设备 `audioOutputType` 选项包含 `LineOut`。
- 默认遍历被测设备支持的 `audioCompressionType`。
- 测试设备需要在 LineOut 端接外置扬声器。
- 录像、停止录像、音频提取、匹配判定逻辑与扬声器测试一致。

## 6. 拾音器测试流程

### 6.1 输入参数

`PickupTestUseCases.run_pickup_test(...)` 主要参数：

- `session`
- `record_channel`
- `record_duration_seconds`
- `send_duration_seconds`
- `similarity_threshold`
- `amplitude_ratio`
- `seed`
- `digit_sequence`
- `fingerprint_source`
- `test_device_input_type`：默认 `MicIn`，LineIn 接入测试传 `LineIn`。
- `test_device_output_type`：默认 `Speaker`。
- `audio_compression_type`：当前轮次播放设备 A 使用的 `audioCompressionType`。

### 6.2 执行步骤

1. 检查参数合法性。
2. 检查 `ffmpeg` 是否可用。
3. 设置测试设备和播放设备 A 的 SDK 对讲模式。
4. 配置测试设备为指定输入和输出，默认 `MicIn + Speaker`。
5. 登录播放设备 A。
6. 配置播放设备 A 为 `MicIn + Speaker + 当前 audioCompressionType`。
7. 尝试开启测试设备复合流音频。
8. 根据播放设备 A 生成测试音。
9. 如果复合流可用，启动测试设备录像。
10. 预录 `2s`。
11. 向播放设备 A 发送 `4s` 测试音，由设备 A 扬声器播出。
12. 播放结束后测试设备继续录制 `3s`。
13. 显式停止保存录像。
14. 显式停止实时预览。
15. 等待 `3s`。
16. 检查录像文件大小。
17. 如果复合流不支持、录像启动失败或录像文件为空，启动测试设备对讲回调 fallback。
18. fallback 路径中，测试设备启动对讲回调，播放设备 A 再播放同一段测试音。
19. 将测试设备对讲回调 PCM 保存为 WAV。
20. 登出播放设备 A。
21. 根据实际来源分析音频：`recording` 或 `talk_callback`。
22. 判断是否有声音。
23. 匹配播放设备 A 的测试音。
24. 输出最终结论。

### 6.3 对讲回调 fallback

触发条件：

- 测试设备复合流能力不支持。
- `NET_DVR_RealPlay_V40` 或保存录像失败。
- 录像文件为空。

fallback 行为：

1. 调用测试设备 `NET_DVR_StartVoiceCom_V30`。
2. 设置 `need_pcm_callback=True`，接收测试设备上传的 PCM 音频。
3. 播放设备 A 播放同一段参考测试音。
4. 将回调 PCM 写入：

```text
pickup_test_talk_callback_audio_<ip>_<audioCompressionType>_<timestamp>.wav
```

5. 使用 `RecordedVideoAnalyzer.analyze_wav_sound_presence(...)` 判断是否有声音。
6. 使用 `RecordedVideoAnalyzer.detect_reference_audio_in_wav(...)` 匹配参考测试音。
7. 最终日志输出 `analysis_source=talk_callback`。

### 6.4 LineIn 接入测试

入口：

```powershell
python .\demo_linein_test.py
```

实现方式：

- 复用 `PickupTestUseCases.run_pickup_test(...)`。
- 默认注入 `test_device_input_type="LineIn"`。
- 能力校验要求测试设备 `audioInputType` 选项包含 `LineIn`。
- 默认遍历播放设备 A 支持的 `audioCompressionType`。
- 同样支持复合流录像优先、对讲回调音频 fallback。
- 测试设备需要在 LineIn 端接外置拾音器。
- 录像、停止录像、音频提取、匹配判定逻辑与拾音器测试一致。

## 7. `audioCompressionType` 遍历和文件命名

默认行为：

1. `demo_speaker_test.py` 和 `demo_lineout_test.py` 登录被测设备后读取被测设备 `audioCompressionType` 选项。
2. `demo_pickup_test.py` 和 `demo_linein_test.py` 登录播放设备 A 后读取播放设备 A `audioCompressionType` 选项。
3. 如果能力中存在多个选项，则逐个切换并执行完整测试。
4. 如果能力中没有 `opt` 选项但能读到当前值，则只执行当前值。
5. 可以通过 `--audio-compression-types` 指定逗号分隔子集。

文件命名：

```text
speaker_test_record_<ip>_<audioCompressionType>_<timestamp>.mp4
speaker_test_reference_<ip>_<audioCompressionType>_<timestamp>_*.wav
speaker_test_record_audio_<ip>_<audioCompressionType>_<timestamp>.wav
pickup_test_record_<ip>_<audioCompressionType>_<timestamp>.mp4
pickup_test_reference_<ip>_<audioCompressionType>_<timestamp>_*.wav
pickup_test_record_audio_<ip>_<audioCompressionType>_<timestamp>.wav
```

文件名中的压缩类型会做安全字符替换，避免出现不能作为文件名的字符。

## 8. 音频分析设计

### 8.1 音频提取

`RecordedVideoAnalyzer.extract_audio(...)` 使用 `ffmpeg` 将录像音频转换为：

- `pcm_s16le`
- `8000Hz`
- 单声道

### 8.2 有声音判断

`analyze_sound_presence(...)`：

1. 按 `20ms` 切帧。
2. 计算每帧 RMS。
3. 任意帧 RMS 大于等于 `300`，则 `has_sound=True`。

### 8.3 参考音频匹配

`detect_reference_audio(...)` 使用多种特征取最高分：

- RMS 能量曲线签名。
- 连续频率存在度评分。
- 主频轨迹相似度评分。
- 参考音频有效发声段裁剪。
- 滑动窗口相关性匹配。
- 时域投影匹配。
- 手动 `--digit-sequence` 兼容模式下的 DTMF 模板搜索。
- 音调偏移容忍搜索。

### 8.4 音调偏移容忍

实验室中被测设备扬声器播放后，录音设备采集到的音调可能发生偏移。算法使用多个 pitch scale 搜索：

```text
0.88, 0.92, 0.96, 1.00, 1.04, 1.08, 1.12, 1.16, 1.20, 1.24
```

这允许识别“音调变高或变低但仍属于当前测试音”的情况。

### 8.5 判定输出

`ReferenceAudioMatchResult` 字段：

- `matched`
- `best_score`
- `best_offset_frames`
- `threshold`
- `expected_digit_sequence`
- `detected_digit_sequence`

说明：

- 默认连续频率音下，`expected_digit_sequence` 和 `detected_digit_sequence` 通常为空。
- `expected_digit_sequence` 是手动 `--digit-sequence` 兼容模式下的期望数字。
- `detected_digit_sequence` 是普通逐帧 DTMF 解码结果，仅辅助观察。
- 最终以综合模板匹配分数 `best_score` 和阈值判定。

## 9. 补光灯测试流程

### 9.1 输入参数

`SupplementLightUseCases.run_supplement_light_test(...)` 主要参数：

- `session`
- `channel`
- `output_dir`
- `settle_seconds`
- `on_threshold`
- `level_threshold`
- `stream_type`

### 9.2 执行步骤

1. 检查 `ffmpeg` 是否可用。
2. 查询补光灯能力和当前配置。
3. 设置补光灯关闭或强度 `0`。
4. 等待 `settle_seconds`。
5. 抓取 baseline 图片。
6. 计算 baseline 图片中心 `40% x 40%` ROI 灰度均值。
7. 设置补光灯开启，强度 `0`。
8. 等待稳定并抓图，计算亮度。
9. 设置补光灯开启，强度 `50`。
10. 等待稳定并抓图，计算亮度。
11. 设置补光灯开启，强度 `100`。
12. 等待稳定并抓图，计算亮度。
13. 计算 `100 - baseline`、`50 - 0`、`100 - 50` 的亮度差。
14. 输出最终结论。

### 9.3 判定逻辑

开启判定：

```text
brightness_100 - brightness_baseline >= on_threshold
```

强度调节判定：

```text
brightness_50 - brightness_0 >= level_threshold
brightness_100 - brightness_50 >= level_threshold
```

最终判定：

```text
passed = light_on_pass and level_pass
```

默认阈值：

- `on_threshold=10`
- `level_threshold=5`

### 9.4 输出文件

默认输出目录：

```text
recordings/supplement_light_tests/<device-ip>/
```

输出文件：

- `supplement_light_test_log_<ip>_<timestamp>.log`
- `supplement_light_baseline_off_<ip>_<timestamp>.jpg`
- `supplement_light_level_0_<ip>_<timestamp>.jpg`
- `supplement_light_level_50_<ip>_<timestamp>.jpg`
- `supplement_light_level_100_<ip>_<timestamp>.jpg`

如果 JPEG 抓图失败并 fallback 到取流抓图，图片扩展名会变为 `_stream.bmp`。
