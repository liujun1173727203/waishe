# 详细设计

## 1. 文件结构

### 1.1 `hikvision_voice.py`

负责 HCNetSDK 的 Python 封装：

- SDK 初始化、日志、清理。
- 设备登录和登出。
- 语音转发。
- 实时预览取流。
- 保存录像文件。
- 显式关闭录像保存和实时预览。

关键类：

- `HikvisionVoiceSDK`
- `DeviceSession`
- `VoiceForwardSession`
- `StreamRecorder`

### 1.2 `hikvision_isapi.py`

负责 ISAPI 配置：

- 查询双向音频能力。
- 设置 MicIn 输入。
- 设置 Speaker 输出。
- 调整麦克风和扬声器音量。
- 查询和开启复合流音频。

关键类：

- `HikvisionIsapiClient`
- `AudioInputCapabilityStatus`
- `TwoWayAudioChannelStatus`
- `CompositeStreamStatus`

### 1.3 `use_cases/voice_talk_cases.py`

负责扬声器测试音生成和发送：

- 根据 `test-tone-id` 生成稳定数字序列。
- 生成 DTMF 双音频率音频。
- 按设备编码格式转为 `G.711u`、`G.711a` 或 `PCM`。
- 按 `20ms` 拆分发送帧。
- 通过语音转发接口发送到被测设备。

### 1.4 `use_cases/speaker_test_cases.py`

负责编排完整扬声器测试：

- 配置被测设备和录音设备。
- 启动录音设备复合流录像。
- 发送被测设备扬声器测试音。
- 停止录像并等待落盘。
- 分析录像音频。
- 输出结构化结果。

### 1.5 `video_analysis.py`

负责录像音频分析：

- 调用 `ffmpeg` 提取音频。
- 判断是否有有效声音。
- 计算参考测试音与录像音频匹配分数。
- 支持音调偏移和环境噪声下的 DTMF 模板搜索。

## 2. SDK 调用设计

### 2.1 初始化和清理

初始化：

1. 加载 `HCNetSDK.dll`。
2. 配置 `HCNetSDKCom` 路径。
3. 配置 OpenSSL 依赖路径。
4. 调用 `NET_DVR_Init`。
5. 可选调用 `NET_DVR_SetLogToFile`。

清理：

1. 清理 Python 侧 callback 引用。
2. 调用 `NET_DVR_Cleanup`。

### 2.2 登录和登出

登录接口：

- `NET_DVR_Login_V40`

登出接口：

- `NET_DVR_Logout`

登出必须在实时预览、保存录像、语音转发关闭之后执行。

### 2.3 语音转发

启动：

- `NET_DVR_StartVoiceCom_MR_V30`

发送：

- `NET_DVR_VoiceComSendData`

停止：

- `NET_DVR_StopVoiceCom`

callback 生命周期策略：

- 启动成功后保存 Python callback 引用。
- SDK stop 成功后再释放 callback 引用。

### 2.4 实时录像

启动：

1. 填充 `NET_DVR_PREVIEWINFO`。
2. 调用 `NET_DVR_RealPlay_V40`。
3. 调用 `NET_DVR_SaveRealData`。
4. 保存实时数据 callback 引用。
5. 等待首帧数据，防止空录像。

关闭：

1. 调用 `NET_DVR_StopSaveRealData`。
2. 调用 `NET_DVR_StopRealPlay`。
3. 释放 callback 引用。
4. 等待 `3s`。
5. 检查录像文件大小。

该关闭顺序经过 `2026-06-02` 实测验证，已同步到所有录像入口。

## 3. ISAPI 配置设计

### 3.1 双向音频能力

能力接口：

- `GET /ISAPI/System/TwoWayAudio/channels/capabilities`

用途：

- 判断是否支持 MicIn。
- 判断是否支持 Speaker。

### 3.2 双向音频通道配置

配置接口：

- `GET /ISAPI/System/TwoWayAudio/channels/<channel>`
- `PUT /ISAPI/System/TwoWayAudio/channels/<channel>`

配置目标：

- `enabled=true`
- `audioInputType=MicIn`
- `audioOutputType=Speaker`
- 麦克风音量最大
- 扬声器音量最大

### 3.3 复合流音频配置

接口：

- `GET /ISAPI/Streaming/channels/<trackStreamID>/capabilities`
- `GET /ISAPI/Streaming/channels/<trackStreamID>`
- `PUT /ISAPI/Streaming/channels/<trackStreamID>`

目标：

- 确认通道支持复合流录音。
- 确保 `Audio.enabled` 或 `Audio.enable` 为 `true`。

## 4. 扬声器测试音设计

### 4.1 生成输入

`VoiceTalkUseCases.prepare_random_audio_file(...)` 支持：

- `duration_seconds`：默认 `4s`。
- `digit_sequence`：可选固定数字序列。
- `fingerprint_source`：测试音 ID，扬声器用例默认传被测设备 IP。
- `seed`：可选随机种子。
- `amplitude_ratio`：振幅比例。

### 4.2 默认测试音

如果未指定 `digit_sequence`，系统根据 `fingerprint_source` 生成 6 位数字序列。生成逻辑使用 SHA-256，保证同一测试音 ID 生成结果稳定，不同测试音 ID 的结果差异较大。

除数字序列外，还会生成稳定的：

- `pitch_scale`：音调比例。
- `gap_ratio`：数字间隔比例。
- `active_ratio`：数字有效发声占比。

这些维度可以帮助区分当前设备的测试音和其他设备同时播放的干扰音。

### 4.3 音频格式

内部生成格式：

- 单声道
- `8000Hz`
- `pcm_s16le`

每个数字使用 DTMF 双音频率合成：

- 低频组：`697`、`770`、`852`、`941`
- 高频组：`1209`、`1336`、`1477`

发送前根据设备当前音频编码转换为：

- `G.711u`
- `G.711a`
- `PCM`

## 5. 扬声器测试流程

### 5.1 输入参数

`SpeakerTestUseCases.run_speaker_test(...)` 主要参数：

- `session`
- `record_channel`
- `voice_channel`
- `record_duration_seconds`
- `send_duration_seconds`
- `similarity_threshold`
- `amplitude_ratio`
- `seed`
- `digit_sequence`
- `fingerprint_source`

### 5.2 执行步骤

1. 检查参数合法性。
2. 检查 `ffmpeg` 是否可用。
3. 设置 SDK 对讲模式为库模式。
4. 检查被测设备 MicIn 能力。
5. 登录录音设备 A。
6. 配置录音设备 A 为 MicIn，并确保复合流携带音频。
7. 配置被测设备为 MicIn 输入和 Speaker 输出。
8. 生成当前被测设备的扬声器测试音。
9. 启动录音设备 A 复合流录像。
10. 等待首帧数据。
11. 预录 `2s`。
12. 向被测设备发送 `4s` 测试音。
13. 播放结束后继续录制 `3s`。
14. 显式停止保存录像。
15. 显式停止实时预览。
16. 等待 `3s`。
17. 检查录像文件大小。
18. 登出录音设备 A。
19. 提取录像音频。
20. 判断是否有声音。
21. 匹配当前被测设备的扬声器测试音。
22. 输出最终结论。

### 5.3 输出结果

`SpeakerTestResult` 字段：

- `login_verified`
- `audio_input_status`
- `two_way_audio_status`
- `recorder_audio_status`
- `composite_status`
- `record_file_path`
- `reference_audio_path`
- `extracted_audio_path`
- `talk_result`
- `sound_result`
- `match_result`

## 6. 音频分析设计

### 6.1 音频提取

`RecordedVideoAnalyzer.extract_audio(...)` 使用 `ffmpeg` 将录像音频转换为：

- `pcm_s16le`
- `8000Hz`
- 单声道

### 6.2 有声音判断

`analyze_sound_presence(...)`：

1. 按 `20ms` 切帧。
2. 计算每帧 RMS。
3. 任意帧 RMS 大于等于 `300`，则 `has_sound=True`。

### 6.3 参考音频匹配

`detect_reference_audio(...)` 使用多种特征取最高分：

- RMS 能量曲线签名。
- DTMF 频率能量签名。
- 参考音频有效发声段裁剪。
- 滑动窗口相关性匹配。
- 期望数字序列模板搜索。
- 音调偏移容忍搜索。

### 6.4 音调偏移容忍

实验室中被测设备扬声器播放后，录音设备采集到的音调可能发生偏移。算法使用多个 pitch scale 搜索：

```text
0.88, 0.92, 0.96, 1.00, 1.04, 1.08, 1.12, 1.16, 1.20, 1.24
```

这允许识别“音调变高或变低但仍属于当前测试音”的情况。

### 6.5 判定输出

`ReferenceAudioMatchResult` 字段：

- `matched`
- `best_score`
- `best_offset_frames`
- `threshold`
- `expected_digit_sequence`
- `detected_digit_sequence`

说明：

- `expected_digit_sequence` 是当前被测设备测试音的期望数字。
- `detected_digit_sequence` 是普通逐帧 DTMF 解码结果，仅辅助观察。
- 最终以综合模板匹配分数 `best_score` 和阈值判定。

## 7. 日志设计

`demo_speaker_test.py` 使用 `TimestampTee` 同步输出到控制台和日志文件。

日志路径：

```text
recordings/speaker_tests/<device-ip>/speaker_test_log_<ip>_<timestamp>.log
```

每行格式：

```text
[YYYY-MM-DD HH:MM:SS] message
```

关键日志：

- `speaker test log file=...`
- `analysis dependency ready ffmpeg=...`
- `prepare speaker validation audio ...`
- `digit sequence=... test_tone_id=...`
- `stop recorder snapshot ...`
- `NET_DVR_StopSaveRealData done`
- `NET_DVR_StopRealPlay done`
- `sound analysis done ...`
- `match analysis done matched=... score=...`
- `final conclusion ...`

## 8. 维护要求

代码变更后需要同步：

1. 更新相关文档。
2. 执行 `python -m py_compile`。
3. 使用真实录像或离线分析命令验证音频分析。
4. 保留日志中的关键判定信息，便于现场排查。
